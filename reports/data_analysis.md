## Реконструкция происхождения датасета

Для обучения бейзлайна использован находящийся в открытом доступе датасет ASL Alphabet, состоящий из $87028$ изображений размера $200$ на $200$. Каждое изображение соответствует одному из $29$ классов, являясь либо одним из $26$ символов алфавита, либо специальным символом (SPACE или DELETE), либо изображением, на котором ничего не показывается (NOTHING). Каждый класс представлен $3000$ фотографий в предложенной авторами обучающей выборке и $1$ фотографии в предложенной авторами тестовой выборке (кроме класса DELETE). В данном проекте все фотографии были объединены в папки, названия которых соответствуют корректным классам, и помещены в общую директорию `data/raw_data`. Разбиение реализовано с помощью скрипта, размещённого в `src/create_split.py`.

По доступному описанию датасета невозможно точно восстановить формальный протокол съёмки, однако подробный анализ позволяет сделать следующие выводы:

$1.$ Вероятнее всего, изображения в датасете $-$ это последовательные кадры одного и того же видео, так как на фотографиях, относящихся к одному классу, наблюдается одна и та же рука в схожих положениях с минимальными изменениями таких условий съёмки как освещённость и фон. Таким образом, датасет представляет собой не выборку независимых изображений, а множество сильно скоррелированных фотографий, полученных в одинаковых условиях.

$2.$ С инженерной точки зрения это означает, что процесс формирования датасета включал выбор ограниченного числа сцен, фиксацию камеры, запись последовательных кадров в процессе исполнения одного и того же жеста и дальнейшее присваивание меток в соответствии с тем символом, который показывался.

$3.$ Разметка данных представляет собой сопоставление каждому кадру одной из $29$ меток, исходя из того, какой символ показывался.

$4.$ Несмотря на идеальный баланс классов, в датасете переизбыточно представлен набор кадров с одним и тем же "исполнителем" для каждого из них. И напротив, полностью исключёнными оказываются кадры с другими "исполнителями", сделанными в иных условиях, включающих расстояние до камеры, освещённость, контрастность фотографий. Это накладывает значительные ограничения на интерпретацию полученных метрик и на обобщающую способность системы.

## Анализ распределений и выявление поддоменов

Для анализа распределений данных были рассмотрены основные технические характеристики изображений (формат и размер), распределение классов внутри выборок, а также простые статистики $-$ яркость, контраст и похожесть изображений (для этого использовалось хэширование и дальнейшее вычисление расстояния между фотографиями).

Анализ показал, что данные стандартизированы по размеру ($200$ на $200$) и формату (`.jpg`), что свидетельствует о заранее нормализованных входных данных и потенциально ограничивает возможность обработки отличных изображений. Распределение по классам ожидаемо идеально равномерное, из-за чего полученные метрики (например, accuracy) не испорчены дисбалансом, однако их интерпретация всё равно должна производиться с учётом сильной корреляции. При этом в каждом классе кроме DELETE есть по одному изображению, формирующему дополнительный поддомен, который не совпадает с основным набором данных.

Внутри каждой из подвыборок также нет дисбаланса $-$ в каждой из них классы представлены практически равномерно, то есть написанный для разбиения данных скрипт сохраняет глобальную структуру данных. Что касается визуальных статистик, анализ показал, что средняя яркость изображений равна примерно $0.507$, средний контраст $0.219$, а средние значения каналов RGB $-$ $[0.519, 0.499, 0.515]$, при этом стандартные отклонения этих величин меньше $0.09$, что указывает на стабильность каждого из этих параметров внутри датасета и подтверждает гипотезу об одинаковых условиях съёмки.

Для всех классов минимальное расстояние между изображениями равно $0$, а средние находятся в диапазоне от $3$ до $9$ (для $64$-битного хэша), то есть внутри каждого класса присутствуют визуально плохо различимые изображения, а различия в среднем между фотографиями очень малы.

Таким образом, можно выделить несколько поддоменов данных:

$1.$ Поддомен "стандартных" для датасета изображений, который составляет большую часть набора. Фотографии для каждого из классов сделаны в одинаковых условиях, с одного ракурса и расстояния. Изображения внутри этого поддомена сильно коррелируют друг с другом.

$2.$ Небольшой дополнительный поднабор изображений $-$ по одной фотографии для каждого класса, снятой в немного отличающихся условиях. При этом фон и композиция кадра остаётся той же.

$3.$ Классы с большим расстоянием между фотографиями внутри. Гипотетически это может быть связано с тем, что для отдельных символов жест является динамическим, а не статическим.
